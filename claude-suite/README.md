# claude-suite вЂ” a comprehensive Claude Code primitive suite for the T3 edge stack

A broad, installable collection of Claude Code **primitives** вЂ” skills, subagents, slash
commands, and hooks вЂ” for building, researching, writing, and operating projects on the
decided **edge T3 stack** (Next.js App Router + Drizzle + Clerk + tRPC + Tailwind v4 + Zod +
RHF). Modeled on the "everything-claude-code" distributions, but specialized to one stack and
anchored to one source of truth so 100+ primitives behave as a single coherent system.

Every primitive points at `CLAUDE.md` (the spine + the **nine inviolable rules**) and composes
with its siblings by name. The suite is **self-growing**: the four meta-skills
(`skill-create`, `agent-create`, `command-create`, `hook-create`) author new primitives in the
exact house style, so it extends without drifting.

- **Software development** вЂ” framework depth (Next.js App Router, Drizzle, tRPC, Clerk, Zod,

## Install

These are standard Claude Code primitives вЂ” copy or symlink them into a `.claude/` directory.

**Project-scoped (recommended):** from your repo root,
```sh
mkdir -p .claude
cp -r /path/to/claude-suite/skills/*   .claude/skills/      # or symlink
cp -r /path/to/claude-suite/agents/*   .claude/agents/
cp -r /path/to/claude-suite/commands/* .claude/commands/
cp -r /path/to/claude-suite/hooks      .claude/hooks
cp /path/to/claude-suite/CLAUDE.md .          # if your repo has no root CLAUDE.md yet
```
Then merge `claude-suite/settings.json` into your `.claude/settings.json` (it wires the hooks
and allowlists the gate scripts). Restart the session so new top-level skill dirs are watched.

**Personal (all projects):** copy the same folders into `~/.claude/{skills,agents,commands}`
and the hooks into `~/.claude/hooks`. Project-scoped primitives override personal ones of the
same name.

> If your repo already has a root `CLAUDE.md`, treat it as canonical and delete the bundled
> copy; point each skill's `source_of_truth` at it.

## Operational notes
- **Frontmatter is the trigger surface.** Only `name` + `description` load at startup; bodies
  and `references/` load on trigger. Descriptions carry the `Use when:` phrases.
- **Scripts run outside context.** `skill-create/scripts/lint-skill.mjs` and the hook scripts
  execute via node; only their output costs tokens.
- **Trust before enabling.** Hooks are executable and run automatically. They make no network
  calls and hold no secrets вЂ” review `hooks/` and `settings.json` before turning them on.

## The baseline-failure gate (read before trusting any skill)

Per the building-skills discipline, **a skill written without a real observed baseline fixes
an imagined problem.** Every `SKILL.md` ships with a `## Baseline failure (REPLACE WITH
OBSERVED TRANSCRIPT)` section written as the failure *class*, labeled as a placeholder. Before
treating a skill as *evaluated*: run its task without the skill, capture what goes wrong,
replace the section with that transcript, then run the dual-session and description-optimizer
checks. The skills are immediately usable as opinionated procedures; this gate is what makes
them evaluated. Use `baseline-capture` to do it.

## Extending the suite

Ask for "a skill / agent / command / hook for X" and the matching meta-skill triggers and
authors it to `docs/house-style.md`. Run `suite-audit` after to lint frontmatter, catch dead
cross-references, and regenerate `docs/composition-map.md`.
