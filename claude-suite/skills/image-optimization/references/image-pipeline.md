Purpose: the per-image decisions — sizing to kill CLS, deriving `sizes` from the grid, format negotiation, `priority` assignment, and the blur/LQIP placeholder choice. This is the markup-side of the pipeline; `next.config` and the CDN loader live in `loaders-and-config.md`.

The discipline: **every content image is sized, the LCP image (and only it) is `priority`, and a
responsive image's `sizes` matches the actual rendered width.** An image that breaks any of these
ships layout shift or oversized bytes that demo fine and fail in the field.

---

## Sizing — the CLS line

An image must reserve its box *before* its bytes arrive, or it pushes content down when it loads
(Cumulative Layout Shift). There are exactly two correct shapes; an unsized `next/image` (or a raw
`<img>`) is neither.

**1. Known intrinsic size → explicit `width` / `height`.** The numbers are the image's intrinsic
aspect ratio, not its rendered size; CSS still scales it. `next/image` uses them to reserve space.

```tsx
import Image from "next/image";
import avatar from "@/assets/avatar.png"; // static import: width/height + blurDataURL inferred

<Image src={avatar} alt="Jane Doe" placeholder="blur" /> // dimensions come from the import
<Image src={remoteUrl} alt="…" width={1200} height={800} /> // remote: you must supply them
```

**2. Fluid / art-directed → `fill` over an aspect-ratio box.** When the rendered size is
layout-driven (a hero, a grid cell), drop explicit dimensions and use `fill`. The **parent** must
own `position: relative` and a reserved ratio (an `aspect-*` utility, or an explicit
`aspect-ratio`), or the image collapses to 0 height.

```tsx
<div className="relative aspect-video">           {/* ratio reserves the space */}
  <Image src={hero} alt="…" fill sizes="100vw" className="object-cover" />
</div>
```

Rule: the aspect-ratio and any wrapper spacing resolve to layout tokens (Rule 3) — that scale is
`layout-composition`'s, not invented here. **Never** a bare `fill` with no sized parent, and never
a content image with neither `width`/`height` nor `fill`.

## `sizes` — derived from the grid, not guessed

For any responsive image (`fill`, or one that scales with the viewport), `sizes` tells the browser
the image's rendered width *per breakpoint, before layout*. The browser uses it to pick a srcset
candidate. **Omit it and the browser assumes `100vw`** and downloads the largest source on every
screen — the quiet LCP regression on mobile.

Read `sizes` straight off the grid (`layout-composition` owns the grid; this reads it):

| Layout | `sizes` |
| --- | --- |
| Full-bleed hero | `100vw` |
| 1 col → 2 → 3 grid (mobile/tablet/desktop) | `(min-width:1024px) 33vw, (min-width:640px) 50vw, 100vw` |
| Fixed content column, image fills it | `(min-width:768px) 768px, 100vw` |
| Sidebar thumbnail | `(min-width:768px) 200px, 50vw` |

The media conditions must match the grid's actual breakpoints. A `sizes` that claims `33vw` while
the CSS renders the cell at `50vw` makes the browser fetch too small (blurry) or too large (waste)
— keep them in lockstep with the columns.

## Format negotiation — configure it, don't assert it

`next/image` negotiates format via the `Accept` header, but **only emits the formats you list**.
The default is WebP only; AVIF (smaller, but slower to encode) ships *only* if you opt in:

```js
// next.config — see loaders-and-config.md for the full block
images: { formats: ["image/avif", "image/webp"] }
```

The original (JPEG/PNG) remains the fallback for browsers that accept neither. The failure to
avoid: a note that says "we get AVIF/WebP" with `formats` left at its default — AVIF never ships.
Set it, then it's true.

## `priority` — exactly one, the LCP image

`priority` makes an image eager and preloaded (`fetchpriority="high"`, no lazy-loading). It is for
the **single** largest above-the-fold image — usually the hero — so the LCP element starts loading
immediately. Every other image stays lazy (the `next/image` default), loading as it nears the
viewport.

- **One** `priority` per route. Spraying it across a gallery preloads everything and there is no
  priority left to give — it competes with the real LCP and regresses it.
- A below-the-fold image with `priority` wastes bandwidth on the critical path for nothing.
- If the LCP image is `priority`, it does **not** need a blur placeholder — it's already inbound.

## Placeholder / LQIP — fill the box before the bytes

A placeholder paints something in the reserved box while the full image loads, so the layout reads
as intentional instead of blank-then-pop.

- **Static imports** → `placeholder="blur"`; Next generates the `blurDataURL` at build, free.
- **Remote images** → you must supply `blurDataURL` (a tiny, ~10–20px base64 generated by `plaiceholder`,
  `sharp`, or the CDN's own tiny-thumbnail transform). Keep it tiny — a large `blurDataURL` is
  inlined into the HTML/JS payload and becomes its own weight.
- **Skip** placeholders for small icons/thumbnails (the swap is imperceptible) and for the
  `priority` LCP image (already preloaded).

Decision: hero and large above-the-fold content imagery → blur. Tiny/utility imagery → none.
Long lists of remote photos → a cheap CDN-generated LQIP, not a per-image build step.
