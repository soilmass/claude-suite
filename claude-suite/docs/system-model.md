# System Model — `claude-suite` (C4 + CRC)

A complete mental model of the suite: what each piece handles, who it communicates with, and
whether each is *deliberate* (its stated responsibility matches what it does, its boundary is
clean, and it is reachable).

## Framework

This model follows the **C4 model** (Simon Brown) — System Context → Containers → Components, at
increasing zoom — augmented at the component level with **CRC cards** (Component · Responsibility ·
Collaborators), which capture exactly "what it handles / who it talks to." Communication uses the
suite's own typed edges, declared in each skill's `## Composes With`:

| Edge | Meaning |
|---|---|
| **Consumes** | depends on the output/setup of another primitive |
| **Feeds** | produces what another primitive inspects/uses |
| **Pairs with** | runs alongside, complementary scope |
| **Runs against** | audits/gates another's output |
| **Hands off** | routes the work onward when it leaves this skill's lane |

The composition graph is auto-listed in `composition-map.md` (regenerated by `suite-audit`). This
file is the *interpreted* model + the deliberateness audit.

## Level 1 — System Context

The suite is a **rule-enforcing playbook** that the Claude main loop loads to build one decided
kind of system.

```
            ┌────────────────────────────────────────────────────────┐
   Developer│  prompts / slash-commands / "spawn the reviewer"        │
   ─────────▶                                                         │
            │                  claude-suite                           │
            │   (skills · agents · commands · hooks · the spine)      │
            │                                                         │
            │  reads ▲ CLAUDE.md + DECISIONS.md  (single source of    │
            │        │ truth — every primitive points here)           │
            └───┬───────────────┬──────────────────┬─────────────────┘
                │ generates      │ gates/guards     │ wires
                ▼                ▼                  ▼
        The Edge App        GitHub + CI        External services
   (Next.js@edge · Drizzle   (the suite's own  (Stripe · Resend ·
    · Clerk · tRPC · Neon/   suite-ci AND the   Upstash · pgvector ·
    Turso · Vercel)          app's gate CI)     analytics, …)
```

**Actors / external systems**
- **Developer** — triggers skills via prompts, `/commands`, or agent spawns.
- **Claude main loop** — loads skills by `description`, follows procedures, spawns agents, fires hooks.
- **The spine** (`CLAUDE.md` + `DECISIONS.md`) — the source of truth every primitive reads; never edited silently (a hook guards it).
- **The Edge App** — the artifact being built (the decided stack).
- **GitHub + CI** — the enforcement surface (the suite gates its *own* repo via `suite-ci`; it also wires the *app's* gate CI).
- **External services** — the integration targets (payments, email, KV, search, analytics).

## Level 2 — Containers (10 functional clusters)

| # | Container | Responsibility (the cluster's one job) | Talks most to |
|---|---|---|---|
| C1 | **Genesis & Spine** | Stand up & protect the non-negotiable foundations once (rails, env, driver, auth, edge rules) | → everything; seeds C2/C3 |
| C2 | **Feature Build-Loop** | Turn a feature description into a complete type-safe, ownership-checked, four-state slice | C1, C3, C4 → C6 |
| C3 | **Data Layer** | Shape, identify, store, evolve, query, and protect data on the edge Drizzle/PG stack | C1, C2 |
| C4 | **Design System** | The `@theme` foundation + token-only styling + image/rich-text rendering + pixel QA | C2, C6, C7 |
| C5 | **Integrations & Platform Services** | Wire external systems & cross-cutting platform concerns into the type chain | C1, C3, C6 |
| C6 | **Quality Gates & Review** | The definition-of-done: 4 gates + judgment passes, each mirrored by a reviewer agent | runs against C2–C5 |
| C7 | **Testing** | Prove the rules hold by allocating each behavior to the cheapest catching layer | C2, C6, C8 |
| C8 | **DevOps, Observability & Cost** | Ship, run, recover, see, and afford the app | C6 → deploy; C1 |
| C9 | **Process, Docs & Decisions** | Project ground truth into durable, sourced artifacts (docs, changelogs, ADRs, research) | reads all; → spine |
| C10 | **Meta / Self-Maintenance** | Keep the suite itself coherent as it grows (author primitives, lint, baseline, enforce) | acts on the suite |

**Primary inter-container flow (the "happy path"):**
`C1 genesis` → `C2 build-loop` (consuming `C3 data` + `C4 design` + `C5 integrations`) →
`C6 gates` → `C7 tests` → `C8 ci/deploy` → `C9 docs/decisions`, with `C10 meta` maintaining the
whole set and `hooks` (in C10) enforcing the spine/rules automatically throughout.

(responsibility matches procedure, boundary clean, reachable); ⚠ = finding (see audit below).

### C6 — Quality Gates & Review — *all ✓*
**Gates (definition of done):** rule-audit · a11y-gate · security-pass · **design-gate** (+ judgment

agent (subjective layer); `perf-auditor` has no gate-skill (perf is a deterministic CI budget).

## Communication model (topology)

- **Hubs (high out-degree):** `stripe-integration` (12), `vertical-slice` (11), `search`/`i18n-l10n`/
  `decision-log` (10) — they compose many foundations. `schema-design` is the biggest **sink**
  (consumed by ~12 C3 skills) — the type-chain root.
- **Foundations (cross-cutting, consumed everywhere):** `schema-design`, `money-modeling`,
  `temporal-data`, `uuidv7-ids`, `design-tokens`, `env-validation`.
- **Leaves (terminal capabilities):** most C5 integrations + the new skills — invoked directly by the
  user, nothing older points forward to them yet. This is expected for a recent cluster, **not** an
  orphan defect: every one is reachable via a trigger phrase and/or command.
- **Pairings:** every gate skill ↔ reviewer agent; `refactor`↔`refactor-executor`;
  `schema-design`↔`schema-architect`; `migration-author`↔`migration-author-agent` — deliberate
  in-loop-vs-isolated-context splits.

## Deliberateness audit

Across ~140 primitives (92 distributable + 14 foundation skills, 17 agents, 25 commands, 8 hooks),
the suite is **overwhelmingly deliberate**: responsibilities match procedures, boundaries are
explicit, and everything is reachable. The findings below are wiring/boundary issues, not
architectural conflict.

**Real findings (worth fixing):**
1. **`trpc-middleware` overreaches `trpc-router-compose`.** Its `description`/`When to Use` claim to
   stand up the context + procedure *builders* (router-compose's job). Intended seam: bases
   (router-compose) vs middleware *bodies* (middleware). *Fix:* retighten the scope text to
   "confirm the context, author the middleware bodies only."
2. **`visual-regression` is miswired into CI.** It hands its job to `ci-pipeline`, but `ci-pipeline`
   never lists it → the snapshot job has no orchestrator. *Fix:* add a visual-regression job (it can
   ride the `playwright-e2e` webServer/storageState).
3. **`ci-pipeline` gate gap.** As the "makes the gates required" surface it omits **`secret-scan`**
   (Rule 9 — a committed key has no build-failing job, only the edit-time hook) and `visual-regression`.
   *Fix:* add both as required jobs.
4. **Measure-range drift.** `design-tokens` says the readable measure ≈ 45–75ch; `layout-composition`
   says ≈ 60–75ch. Ownership is coordinated but the numbers diverge. *Fix:* one canonical range in
   `design-tokens` (now the type owner); `layout-composition` references it.
5. **`doc-writer` ↔ `readme-author` boundary unstated.** The agent claims README authorship but only
   names api-docs/technical-writing as hand-offs. *Fix:* add a line naming `readme-author` (and
   `changelog-from-commits`) as the procedures it executes.
6. **`suite-audit` has no command** and `/audit` is taken by `rule-audit`. *Fix:* add `/suite-audit`
   (or `/lint-suite`) so self-maintenance is as reachable as the code gates.

**Advisory (graph completeness / cohesion, optional):**
- `t3-genesis` `Composes With` omits the 4 seams it seeds (env-validation, neon-turso-driver,
  clerk-auth-flows, trpc-router-compose) — the back-edges exist; add the forward ones.
- `load-and-mutation-testing` fuses load + mutation testing (one primitive, two responsibilities) —
  split if either grows.
- The four-state (Rule 4) property is asserted at 4 test layers with no single owner — add one line
  to `test-strategy` assigning the per-component layer.
- No `mcp-create` meta-skill for `.mcp.json` entries — add it or scope MCP out of `skill-create` explicitly.
- `/commit` and `/research` target external (global/plugin) skills — fine in this environment; a
  portability note if the suite is ever distributed fully standalone.

**Discarded false positives:** `perishable-refresh` and `refactor` were flagged as "missing from
`claude-suite/skills/`" — both are **foundation skills**, `.claude`-only **by design** (like
rule-audit, a11y-gate, design-tokens). Not drift.
